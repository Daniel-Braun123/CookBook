CookBook

Technische Projektdokumentation — Full-Stack Rezeptverwaltung

Projektname

CookBook — Eine moderne Webanwendung zur Verwaltung, Entdeckung und Bewertung von Rezepten.

Zweck

CookBook ermöglicht es Benutzern, Rezepte zu erstellen, zu durchsuchen, zu bewerten und zu speichern. Die Anwendung bietet sowohl eine lokale Registrierung als auch Google OAuth2 Login. Administratoren können Benutzer und Inhalte verwalten.

Hauptfunktionen

  • Rezeptverwaltung — Erstellen, Bearbeiten, Löschen von Rezepten mit Zutaten, Kochschritten und Nährwertinformationen
  • Bewertungssystem — 1–5 Sterne Bewertungen mit Kommentaren und aggregiertem Rating
  • Rezeptsuche — Volltextsuche über Titel und Beschreibung
  • Kategoriefilter — Rezepte nach Kategorien filtern
  • Gespeicherte Rezepte — Benutzer können Rezepte als Favoriten speichern
  • Benutzerverwaltung — Registrierung, Login (lokal + Google OAuth2), Profilbearbeitung
  • Admin-Panel — Benutzerverwaltung, Rollenzuweisung, Inhaltsmoderation

Technische Zusammenfassung

FrontendAngular 17.3 (Standalone Components) + Tailwind CSS + Angular Material
BackendSpring Boot 4.0.1 (Java 17) — RESTful API
DatenbankPostgreSQL mit Flyway Migrations (8 Tabellen)
AuthentifizierungJWT + Google OAuth2
BuildGradle (Backend), Angular CLI / Bun (Frontend)

2. Systemarchitektur

Das Projekt folgt einer klassischen 3-Schichten-Architektur mit strikter Trennung zwischen Frontend (SPA), Backend (REST API) und Datenbank.

Kommunikationsfluss

Angular SPA:4200
→ HTTP/JSON →
Spring Boot API:8080
→ JPA →
PostgreSQL:5433

Das Frontend kommuniziert ausschließlich über REST-API-Aufrufe mit dem Backend. JWT-Tokens werden im Authorization Header als Bearer-Token mitgesendet. Bild-Uploads gehen direkt vom Frontend zu Cloudinary.

3. Verwendete Technologien

Spring Boot 4.0.1

Java-basiertes Backend-Framework. Stellt die REST API bereit, verwaltet Dependency Injection, Security und Datenbankanbindung.

Angular 17.3

TypeScript-basiertes Frontend-Framework mit Standalone Components. Verantwortlich für die gesamte UI und Client-seitige Logik.

PostgreSQL

Relationale Datenbank. Speichert alle Benutzer-, Rezept-, Bewertungs- und Kategorie-Daten in 8 Tabellen.

Hibernate / JPA

ORM-Framework für die Datenbankabstraktion. Entity-Mapping, Lazy Loading, Query-Derivation und native Queries.

Flyway

Datenbankmigrationen (V0.0 – V0.7). Versioniert Schema-Änderungen und ermöglicht reproduzierbare Deployments.

JWT (jjwt 0.12.5)

JSON Web Tokens für zustandslose Authentifizierung. HMAC-SHA256-Signierung, 24h Gültigkeit.

Spring Security + OAuth2

Absicherung aller Endpunkte. Google OAuth2 Login mit Custom Success/Failure Handlers.

Gradle

Build-Tool für das Backend. Dependency-Management, Kompilierung, Tests und JAR-Erstellung.

Tailwind CSS

Utility-First CSS Framework. Responsives Design mit Custom Color Palette (Terracotta, Sage, Ivory).

Angular Material

UI-Komponenten-Bibliothek. Dialoge, Icons und Buttons für konsistente UX.

Cloudinary

Cloud-basierter Bild-Upload-Service. Validierung auf 5MB Dateigröße im Frontend.

BCrypt

Passwort-Hashing-Algorithmus. Sicheres Speichern von Benutzerpasswörtern mit konfigurierbarer Stärke.

Datenfluss: Rezept erstellen

// POST /api/recipes/create
@AuthenticationPrincipal Users currentUser
    → RecipeService.createRecipe(request, currentUser)
        → categorieRepository.findByName(categoryName)   // Kategorie auflösen
        → recipeRepository.save(recipe)                    // Rezept speichern
        → ingredientsRepository.saveAll(ingredients)       // Zutaten speichern
        → cookingStepsRepository.saveAll(steps)            // Schritte speichern
        → nutritionInfoRepository.save(nutrition)          // Nährwerte speichern
    → RecipeResponseDto.fromEntity(recipe)                // Entity → DTO

Datenfluss: Bewertung abgeben

// POST /api/reviews/recipe/{recipeId}
    → ReviewService.submitReview(recipeId, user, request)
        → Validierung: Rezept existiert, Selbstbewertung verhindert, Sterne 1-5
        → Review erstellen oder aktualisieren (Upsert)
        → updateRecipeAggregate(recipe)                   // AVG + COUNT berechnen
            → reviewRepository.calculateAverageRating()
            → reviewRepository.countByRecipeId()
            → recipe.setRating() + recipe.setReviewCount()
    → ReviewResponseDto.fromEntity(review)

Dependency Injection

Alle Abhängigkeiten werden per Constructor Injection aufgelöst — der empfohlene Ansatz in Spring Boot. Die Injection-Kette verläuft strikt top-down:

Controller → Service → Repository → JPA EntityManager → DataSource

Exception Handling

Das Projekt verwendet try-catch-Blöcke in den Controllern mit manueller ResponseEntity-Erstellung. Es gibt keinen globalen @ControllerAdvice — ein Verbesserungspotenzial (siehe Abschnitt 9).

6. Datenbank

Übersicht

PostgreSQL auf Port 5433, Schema wird durch 7 Flyway-Migrationen (V0.0 – V0.7) verwaltet. ddl-auto: none — kein automatisches Schema-Management durch Hibernate.

Entity-Relationship-Diagramm

Die vollständige draw.io-Datei liegt unter database-model.drawio im Projektverzeichnis.

Tabellen-Details

users

Speichert alle Benutzerkonten — sowohl lokale (Passwort) als auch OAuth2-Benutzer (Google). Das Feld password ist nullable für OAuth-Benutzer.

Rollen: USER (Standard), ADMIN — gesteuert über role-Spalte.

Auth Provider: LOCAL, GOOGLE

recipes

Zentrale Tabelle mit Rezeptdaten. Enthält aggregierte Werte (rating, review_count) die bei jeder Bewertungsänderung aktualisiert werden. Tags werden als JSONB-Array gespeichert.

Schwierigkeit: EINFACH, MITTEL, SCHWER

reviews

Bewertungen mit 1–5 Sternen und optionalem Kommentar. Ein Benutzer kann ein Rezept nur einmal bewerten (Unique Constraint). CASCADE-Löschung bei Rezept- oder User-Löschung.

saved_recipes

Junction-Tabelle für die M:N-Beziehung zwischen Benutzern und ihren gespeicherten Rezepten. Ermöglicht die „Favoriten"-Funktion.

9. Performance & Verbesserungsvorschläge

✓ Positive Architekturentscheidungen

  • Flyway-Migrationen — Reproduzierbares, versioniertes Datenbankschema
  • DTO-Pattern — Saubere Trennung zwischen Entity- und API-Schicht
  • Zustandslose Authentifizierung — JWT ermöglicht horizontale Skalierung
  • Aggregiertes Rating — Vermeidet teure JOIN-Queries bei jedem Rezept-Abruf
  • Constructor Injection — Best Practice für Dependency Injection in Spring
  • Standalone Components — Modernes Angular-Pattern ohne NgModules
  • saved-ids Endpoint — Effiziente ID-Liste statt vollständiger Rezept-Daten für Frontend-Checks

✗ Verbesserungspotenzial

  • Kein globaler ExceptionHandler — Fehlt ein @ControllerAdvice für einheitliche Fehlerantworten. Aktuell try-catch in jedem Controller.
  • Hardcoded Credentials — Datenbank-Passwort in application.properties. Sollte über Umgebungsvariablen konfiguriert werden.
  • N+1 Query Risiko — Lazy-Loading von Author/Category bei Listen-Abfragen. Kann mit @EntityGraph oder JOIN FETCH optimiert werden.
  • Keine Pagination — getAll liefert alle Rezepte ohne Limit. Bei wachsender Datenmenge sollte Pageable verwendet werden.
  • Keine Input-Validierung — Fehlende Bean Validation (@Valid, @NotBlank) auf DTOs. Validierung nur programmatisch in Services.
  • Typo in Endpoint-Pfad — /api/ingridients statt /api/ingredients
  • Keine Rate Limiting — Kein Schutz vor Brute-Force-Angriffen auf Login-Endpoint
  • CORS Hardcoded — Localhost-URL in Produktion nicht verwendbar, sollte konfigurierbar sein
